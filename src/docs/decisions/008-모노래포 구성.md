모노레포로 뽑을 수 있는 것들
packages/types/ ← entities/bookmark/model 의 타입들
packages/ui/ ← shared/ui (icons, input)
packages/config/ ← tsconfig, eslint
나머지 (features, widgets, pages) 는 앱 종속이라 apps/web/ 안에 그대로 두기로 했다.

## pnpm 으로 정한 이유

1. content-addressable store
   같은 패키지는 디스크에 딱 한 번만 저장해요. 프로젝트 10개에서 React 쓰면 npm은 10번 설치, pnpm은 1번 저장하고 나머지는 하드링크로 연결해요. 모노레포에서 패키지가 많아질수록 이게 엄청난 차이가 난다.

- 하드링크
  하드링크는 다른 이름으로 같은 데이터를 가리키는 것을 말한다.
  ```
  프로젝트A/node_modules/react  ─┐
                             ├──→ 디스크의 실제 react 데이터 (1개)
  프로젝트B/node_modules/react  ─┘
  ```

2. 엄격한 의존성 구조
   npm은 node_modules가 평탄화(hoisting)돼서 내가 설치 안 한 패키지도 접근 가능해요. 이게 유령 의존성(phantom dependency) 문제를 만들어요. pnpm은 이걸 막는다.

A 설치할 때 B가 딸려 오는 건 맞아요. 근데 유령 의존성의 핵심은 딸려온 B를 내 코드에서 직접 import해서 쓸 수가 있다.

```
내 package.json에 B 없음
// 근데 이게 동작해버림
import { something } from 'B'
```

이게 왜 문제냐면, 나중에 A가 B 의존성을 끊거나 버전 바꾸면 내 코드가 이유도 모르고 터져요. 내가 B를 명시적으로 관리한 적이 없으니까.

```
// npm - 평탄화돼서 B가 최상위에 있음아
node_modules/
  A/
  B/   ← 내가 설치 안 했는데 접근 가능

// pnpm - 심볼릭링크로 격리
node_modules/
  A/   ← 내가 설치한 것만
  .pnpm/  ← B는 여기 격리됨
```

그래서 pnpm은 package.json에 명시 안 한 패키지는 import하면 에러 터진다.
이렇게 되면 쓰고 있는 라이브러리 버전이 변해서 쓰지 않는다거나 했을때에 대한 에러에 대비 할 수 있다.

## TypeScript 설정 공유 전략

모노레포 내의 여러 패키지와 앱에서 일관된 TypeScript 규칙을 유지하고 설정을 중복하지 않기 위해 `packages/typescript-config`를 생성하여 설정을 패키지화했다.

### 1. 구조 (Configuration Hierarchy)

- **packages/typescript-config/base.json**: 모든 패키지의 근간이 되는 기본 설정 (Strict 모드, ESM 등).
- **packages/typescript-config/nextjs.json**: Next.js 프로젝트를 위한 특화 설정 (jsx, plugins, lib 등). `base.json`을 확장하여 구성.
- **apps/web/tsconfig.json**: 실제 앱에서 위 설정들을 불러와 프로젝트 루트에 맞게 최종 적용.

### 2. 설정 예시 (Code Snippets)

#### **packages/typescript-config/base.json**

```json
{
  "$schema": "https://json.schemastore.org/tsconfig",
  "compilerOptions": {
    "target": "ESNext",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "skipLibCheck": true,
    "isolatedModules": true,
    "esModuleInterop": true,
    "incremental": true
  }
}
```

#### **packages/typescript-config/nextjs.json**

```json
{
  "extends": "./base.json",
  "compilerOptions": {
    "jsx": "preserve",
    "lib": ["dom", "dom.iterable", "esnext"],
    "allowJs": true,
    "noEmit": true,
    "plugins": [{ "name": "next" }]
  }
}
```

#### **apps/web/tsconfig.json**

```json
{
  "extends": "@sb/typescript-config/nextjs.json",
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  },
  "include": ["next-env.d.ts", "**/*.ts", "**/*.tsx", ".next/types/**/*.ts"],
  "exclude": ["node_modules"]
}
```

패키지화를 해야지 되기 때문에

```json
{
  "name": "@smart-bookmark/typescript-config",
  "version": "0.0.0",
  "private": true,
  "license": "MIT"
}
```

이런식으로 구현을 함
여기서 private는 true를 함 -> 내부에서 쓰는 것이기 때문에 npm에 베포를 할 필요가 없음

### 3. 기대 효과

- **일관성**: 모든 패키지가 동일한 엄격도(Strict) 규칙을 따름.
- **유지보수성**: 공통 규칙 변경 시 `base.json` 한 곳만 수정하면 전체 워크스페이스에 반영됨.
- **가독성**: 개별 앱의 `tsconfig.json`이 매우 간결해짐.

## 모노레포의 장단점

장점

공통 코드 한 번 수정하면 모든 앱에 즉시 반영
타입, 유틸, 컴포넌트 공유가 쉬움
하나의 PR로 여러 앱 동시 수정 가능
의존성 버전 통일 관리 가능

단점

레포 크기가 커질수록 clone, CI 속도 느려짐
팀 분리돼 있으면 권한 관리 복잡
초기 설정 복잡도 높음
빌드 캐싱 제대로 안 하면 오히려 빌드 느려짐

## 모노래포가 맞는 상황 vs 안맞는 상황

모노레포가 맞는 상황

공유 코드가 많을 때
앱들이 같은 백엔드, 같은 타입 쓸 때
혼자 또는 작은 팀

멀티레포가 맞는 상황

앱 간 공유 코드가 거의 없을 때
팀이 완전히 분리돼서 배포 주기도 다를 때
각 앱의 기술 스택이 완전히 다를 때
