# ADR 000: 견고한 프로젝트 초기 설정 (Node 고정, 품질 자동화 및 모노레포 전략)

## 1. 개요 (Context)

다양한 개발 환경(OS, Node 버전 등)에서 발생할 수 있는 환경 차이를 제거하고, 코드 품질을 일관되게 유지하며, 프로젝트 규모 확장에 유연하게 대응하기 위해 프로젝트 초기부터 강력한 환경 제약과 자동화 도구를 도입한다.

## 2. 설계 결정 (Design Decisions)

### 2.1. 런타임 및 패키지 매니저 고정 (Environment Isolation)

프로젝트의 안정성을 위해 개발 런타임 버전을 엄격히 강제한다.

- **Node.js 고정**: `.nvmrc` 파일을 통해 `24.14.0`으로 고정하여 팀원 간 실행 환경을 일치시킨다.
- **패키지 매니저 고정**: `packageManager: pnpm@10.30.2`를 명시하여 패키지 설치의 결정론적(Deterministic) 결과물을 보장한다.
- **엄격성 부여**: `.npmrc`의 `engine-strict=true` 설정을 통해 지정된 버전이 아닐 경우 설치 자체가 차단되도록 한다.

### 2.2. 코드 품질 관리 자동화 (Code Quality Automation)

수동 리뷰 없이도 기본적인 코드 스타일과 품질이 보장되도록 도구를 구성한다.

- **ESLint (Static Analysis)**: Next.js(Core Web Vitals), TypeScript, Storybook 규칙이 통합된 Flat Config를 사용하여 런타임 오류 가능성을 사전 차단한다.
- **Prettier (Formatting)**: `prettier-plugin-tailwindcss`를 포함하여 Tailwind 클래스 정렬까지 자동화함으로써 가독성을 극대화한다.
- **Storybook (UI Isolation & Documentation)**:
  - UI 컴포넌트를 비즈니스 로직과 분리하여 독립적으로 개발하고 테스트할 수 있는 환경을 구축한다.
  - 컴포넌트의 가시적인 상태를 문서화하여 팀 내 디자인 시스템 공유 및 협업 효율을 높인다.
- **Husky & lint-staged (Gatekeeping)**:
  - `pre-commit` 훅을 통해 커밋 전 변경된 파일에 대해서만 Lint와 Format 체크를 수행한다.
  - 품질 기준을 통과하지 못한 코드는 아예 커밋될 수 없도록 강제한다.

### 2.3. 모노레포(Monorepo) 아키텍처 채택

FSD(Feature-Sliced Design) 패턴의 효율성을 극대화하기 위해 `pnpm workspace` 기반의 모노레포 구조로 전환한다.

- **네임스페이스**: 모든 내부 패키지는 `@sb/` (Smart Bookmark) 접두사를 사용한다.
- **패키지 분리**: `@sb/ui`(공통 UI), `@sb/shared`(유틸), `@sb/entities`(도메인 로직), `@sb/configs`(설정 패키지)로 나누어 재사용성을 높인다.

## 3. 판단 기준 및 기대 효과 (Rationale)

1.  **환경 일관성**: 신규 투입 인원도 `.nvmrc`와 `pnpm install`만으로 즉시 빌드 가능한 환경을 갖춘다.
2.  **기술 부채 방지**: 커밋 시점에 품질 검사를 수행함으로써 스타일 문제나 사소한 버그가 저장소에 유입되는 것을 원천 차단한다.
3.  **유연한 확장성**: `Backend Agnostic` 원칙에 따라 설계된 `entities` 패키지는 추후 백엔드 변경이나 모바일 앱 확장 시 핵심 로직을 재사용할 수 있게 한다.

## 4. 향후 과제

- `apps/` 및 `packages/` 폴더 구조 실제 생성 및 코드 마이그레이션.
- 공통 설정(`ESLint`, `TypeScript`)을 `packages/configs`로 추출하여 전역 패키지 관리.
