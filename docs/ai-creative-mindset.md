# 창의적 AI 자동화를 위한 사고방식

> 작성일: 2026-04-04

---

## 핵심은 하나다

**AI를 도구로 보지 말고, 협업 상대로 보는 것.**

대부분 개발자가 AI를 이렇게 쓴다:

```
내가 설계한 기능 → AI한테 코드 시켜
```

창의적 자동화는 이렇게 된다:

```
문제 상황 → AI가 어떻게 인식할까 → 어떤 컨텍스트를 주면 스스로 판단할까
```

차이는 **"내가 로직을 짜느냐"** vs **"AI가 판단할 환경을 짜느냐"** 다.

이 프로젝트에서 가장 창의적인 부분이 뭔지 알아?
→ `CLAUDE.md`에 개발 프로세스 전체를 적어놓고, AI가 Notion 상태를 읽어서 스스로 개발 흐름을 따르게 한 것.
이게 바로 **"AI가 판단할 환경 설계"** 다.

---

## 사고 전환 3가지

### 1. "트리거 → 판단 → 행동" 루프를 찾는 훈련

일상에서 반복되는 작업을 볼 때 이 질문을 습관화해:

> **"이 작업에서 사람이 판단하는 부분은 어디고, 그 판단 기준은 명문화될 수 있나?"**

예시:

- 코드 리뷰할 때 → "이 PR이 머지 가능한지" 판단 기준이 뭔가 → 명문화 → AI에게 위임 가능
- 북마크 태그 달 때 → "이게 React 글인지" 판단 기준이 뭔가 → 이미 AI가 하고 있음

명문화될 수 없는 판단은 사람이, 명문화된 판단은 AI가 하면 된다.

---

### 2. 프롬프트보다 "컨텍스트 엔지니어링"이 중요하다

프롬프트를 잘 쓰는 것보다 **AI에게 어떤 정보를 어떤 구조로 줄지**가 훨씬 중요하다.

**지금 코드의 아쉬운 부분:**

```typescript
// 본문 2000자를 통째로 던짐
const bodyText = bodyChunks.join(" ").slice(0, 2000);
```

**더 좋은 방식 — 판단하기 좋은 구조로 설계:**

```typescript
const context = {
  url: "출처 도메인 (신뢰도 힌트)",
  ogTitle: "의도된 제목",
  ogDescription: "저자가 쓴 요약",
  firstParagraph: "본문 첫 단락 (핵심 주제)",
  headings: ["H2들 목록 (글 구조)"],
  codeLanguages: ["typescript", "react"], // 기술 힌트
};
```

**정보의 양보다 정보의 구조**가 AI 출력 품질을 결정한다.

---

### 3. 실패를 설계하기

보통 자동화할 때 "성공 케이스"만 설계하고 실패하면 당황한다.
AI 자동화는 반드시 실패한다. 창의적인 설계는 **실패했을 때 어떻게 되는지**를 처음부터 설계하는 것이다.

이 프로젝트 `CLAUDE.md`에 이미 있다:

```
❌ 5회 초과 → 상태: 검수요청, 댓글에 실패 로그 기록
```

이게 맞는 방향이다. 더 확장하면:

- AI 실패 → 어떤 컨텍스트가 부족했는지 자동 분석
- 같은 패턴이 3번 반복되면 → CLAUDE.md Rules에 자동 추가

## 지금 당장 해볼 수 있는 실험

코드 짜는 것보다 이게 훨씬 빠르게 사고방식을 바꿔준다.

| 주차  | 할 것                                                     |
| ----- | --------------------------------------------------------- |
| 1주차 | 하루 한 가지 반복 작업을 찾아서 "판단 기준"을 글로 써보기 |
| 2주차 | 그 판단 기준으로 프롬프트 만들고 실제로 시켜보기          |
| 3주차 | 실패 케이스 분석 → 컨텍스트 구조 개선                     |
| 4주차 | 트리거 자동화 (시간, 이벤트, 상태 변화로 자동 실행)       |

## 결론

지금 가장 빠르게 성장하는 방법은 새 기술을 배우기 전에,
**현재 프로젝트에서 "사람이 수동으로 하고 있는 판단"을 찾아서 AI에게 위임하는 실험을 계속 하는 것.**

아키텍처는 이미 좋다. 지금 단계는 사고 근육을 키우는 것이다.
