# AI 개발 환경 (팀 이식 가능)

이 레포는 개인 IDE 설정이 아니라, **클론하면 팀 전체가 동일하게 쓰는 AI 개발 규약**을 코드로 커밋해 둔다.
"AI를 써봤다"가 아니라 **"무엇을 자동화하고 무엇을 사람 판단에 남길지 설계했다"**가 이 문서의 요지다.

## 설계 원칙: 결정론 경계

무엇을 강제(자동)하고 무엇을 온디맨드(사람 호출)로 남길지는 **하나의 기준**으로 나눈다 —

> **빠르고 · 객관적이고 · 결정론적인 것만 강제한다. 느리고 · 주관적이고 · 비싼 판단은 사람이 부를 때만 돈다.**

모든 걸 자동화하면 느려지고, 개발자가 우회하기 시작하면 자동화는 죽는다.

| 시점           | 수단                        | 대상                                           | 성격               |
| -------------- | --------------------------- | ---------------------------------------------- | ------------------ |
| 파일 저장 시   | **hook** (`PostToolUse`)    | prettier + eslint --fix                        | 강제 · 결정론      |
| 커밋/PR 게이트 | **CI required check**       | typecheck · test · build · **FSD 레이어 경계** | 강제 · 결정론      |
| 필요할 때      | **서브에이전트 (온디맨드)** | 코드리뷰 · 보안 · FSD 판단 · QA · UX           | 사람 호출 · 비결정 |
| 태스크 진행    | **MCP 파이프라인**          | Notion 상태 기반 개발 흐름                     | 트리거             |

핵심: **AI 코드리뷰는 일부러 hook에 넣지 않았다.** LLM 호출은 느리고 비결정적이라
매 커밋 강제 시 개발자가 우회하게 된다. 리뷰는 사람이 판단할 시점에 부른다.

## 1. 커스텀 서브에이전트 (`.claude/agents/`)

범용 "리뷰해줘"가 아니라 **이 프로젝트 도메인 지식이 프롬프트에 박제된** 재사용 에이전트.
레포에 커밋돼 있어 팀원 누구나 동일한 기준으로 호출한다.

| 에이전트                    | 역할                       | 프로젝트 특화 지점                                                              |
| --------------------------- | -------------------------- | ------------------------------------------------------------------------------- |
| `code-review`               | 구조·타입·성능 공격적 리뷰 | FSD 레이어 위반, `getUser` vs `getSession`, pgvector 인덱스, TanStack staleTime |
| `security-reviewer`         | 공격자 시점 취약점         | `validateSsrf.ts`, `is_guest` 쿠키 신뢰, Service Role Key 노출, RLS             |
| `fsd-checker`               | 레이어 의존성 방향         | `pages→…→shared` 역방향 import, 슬라이스 간 참조                                |
| `qa-reviewer` / `ux-critic` | 완성도·인터랙션            | 깜빡임, 엣지 케이스, 접근성                                                     |

호출: `리뷰해줘 [대상]`, `보안 검토해줘 [대상]`, `FSD 검토` 등 (CLAUDE.md 트리거).

## 2. 컨텍스트 엔지니어링

AI가 매번 코드베이스를 헤매지 않도록 **탐색 비용을 구조로 낮춘다.**

- **`PROJECT_INDEX.md`** — 키워드 → 파일 맵. Grep 남발(컨텍스트 오염) 대신 핀포인트 탐색.
  탐색 규칙(CLAUDE.md): INDEX에 없으면 임의 Glob/Grep 금지 → AI가 없는 사실을 지어내는 걸 차단.
- **`CLAUDE.md`** — 아키텍처·코드 스타일·작업 규칙·트리거를 항상 로드되는 단일 소스로 고정.
- **`docs/decisions/`** — "왜 이렇게 했는가"를 증상→추적→수정→교훈으로 기록. AI가 과거 결정을 재현.

## 3. hook — 저장 시 포맷·린트 강제

AI가 Write/Edit로 파일을 건드릴 때마다 prettier + eslint --fix를 **강제**한다
→ PR에서 스타일 지적이 사라지고 리뷰가 로직에만 집중된다. (결정론적이라 hook에 적합한 대표 사례)

- **로직**: `.claude/hooks/format.mjs` (커밋됨 → 팀 공유). hook 입력을 stdin JSON으로 받아
  `tool_input.file_path`를 파싱하고, 포맷 가능한 확장자면 prettier·eslint를 돌린다.
  포맷 실패로 편집을 막지 않는다(항상 exit 0).
- **wiring**: `.claude/settings.json`
  ```jsonc
  "hooks": { "PostToolUse": [{ "matcher": "Write|Edit",
    "hooks": [{ "type": "command", "command": "node .claude/hooks/format.mjs" }] }] }
  ```

> 설계 메모: 로직을 `.mjs` 스크립트로 분리한 이유 — `settings.json`은 `.gitignore` 대상(민감정보
> 정책)이라 팀에 공유되지 않는다. 커밋되는 스크립트에 로직을 두면 팀 전체가 동일 포맷 규칙을
> 공유하고, 팀원은 wiring 한 줄만 자신의 `settings.json`에 넣으면 된다.
> (과거엔 이 hook이 오타 파일 `setting.json`에 있었고 존재하지 않는 `$CLAUDE_FILE_PATH`를
> 참조해 실제로는 돌지 않았다 — 실측으로 발견해 stdin 파싱으로 교정.)

## 4. MCP 태스크 파이프라인 (Notion)

세부 태스크를 Notion DB(태스크명/상태/검수)로 관리하고,
`방향성 → 개발중 → 테스트중 → 완료` 상태 전이를 AI가 트리거받아 진행한다.
상세: `.claude/skills/task-pipeline/`.

## 효과 (관찰 가능한 형태로)

- **탐색**: PROJECT_INDEX 도입으로 파일 탐색이 Grep 스캔 → 핀포인트로 전환, 컨텍스트 토큰 절감.
- **스타일**: 포맷 hook 도입 후 PR 리뷰에서 스타일 코멘트 소멸 → 로직 리뷰에 집중.
- **일관성**: 리뷰 기준이 사람 머릿속이 아니라 커밋된 에이전트 프롬프트에 있어 **누가 돌려도 같은 기준.**
- **이식성**: 클론 즉시 동일 환경 → 개인 생산성이 아니라 **팀 생산성 곱하기.**
