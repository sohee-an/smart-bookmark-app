# 웹폰트 도입: 통파일 vs 동적 서브셋 실측 비교

- **날짜**: 2026-06-07
- **상태**: 적용 완료 (동적 서브셋 채택)
- **태그**: 웹폰트, 성능, 한글 서브셋, unicode-range

---

## 1. 배경

- 기존 상태: `layout.tsx`가 next/font로 Geist(영문 전용)를 로드했지만 `@theme`에 `--font-sans` 연결이 없어 **다운로드만 되고 적용되지 않는 죽은 코드**였음 (`font-mono` 사용처 0곳)
- 실제 렌더링: 시스템 폰트 혼합 (영문 Segoe UI/SF Pro + 한글 맑은 고딕/Apple SD Gothic Neo) — 한 문장 안에서 두 폰트가 섞여 자간·두께가 미세하게 어긋남
- 목표: 한·영 통일된 한글 폰트(Pretendard) 도입 + **통파일과 동적 서브셋의 비용 차이를 실측으로 검증**

## 2. 실험 설계

같은 페이지를 같은 조건(dev 모드, Disable cache)에서 두 방식으로 측정:

| 단계  | 방식            | 구성                                                                                                |
| ----- | --------------- | --------------------------------------------------------------------------------------------------- |
| 1단계 | **통파일**      | `public/fonts/PretendardVariable.woff2`(2MB) + 수동 `@font-face`(`font-display: swap`)              |
| 2단계 | **동적 서브셋** | `pnpm add pretendard` → `pretendardvariable-dynamic-subset.css` import (unicode-range 조각 수백 개) |

공통: `@theme`의 `--font-sans`에 5칸 폰트 스택 — 웹폰트 → 시스템 → 한글 폴백 명시 → generic

```css
--font-sans:
  "Pretendard Variable", ui-sans-serif, system-ui, "Apple SD Gothic Neo", "Malgun Gothic",
  sans-serif;
```

## 3. 실측 결과 (DevTools Network)

|                         | 1단계: 통파일          | 2단계: 동적 서브셋           | 변화     |
| ----------------------- | ---------------------- | ---------------------------- | -------- |
| 폰트 요청 수            | 1건                    | **17건** (조각, 각 21~38 kB) |          |
| **폰트 전송량**         | **2,058 kB**           | **457 kB**                   | **−78%** |
| 페이지 전체 transferred | ~3,100 kB              | 1,957 kB                     | −37%     |
| 폰트의 페이지 비중      | 66% (단일 최대 리소스) | 23%                          |          |

### 해석

- woff2는 이미 최고 압축 포맷이라 **포맷으로는 더 못 줄임** — 한글 폰트(글리프 11,172자)의 용량 문제는 "글자 수를 줄이는" 서브셋으로만 해결됨
- 동적 서브셋 = 폰트를 unicode-range 단위 조각으로 쪼개고, **페이지에 실제 등장한 글자의 조각만 다운로드**. 화면 글꼴 품질·한영 통일은 통파일과 완전히 동일
- 457 kB가 일반 권장 예산(100~300 kB)보다 다소 높은 건 대시보드에 다양한 글자(북마크 제목들)가 렌더링되어 조각 17개가 호출됐기 때문 — 필요 시 정적 굵기 서브셋 등 추가 최적화 여지 있음

## 4. 곁가지 수정 (실험 중 발견한 버그)

**미들웨어 매처가 정적 자산을 인증 체크로 끌고 가던 문제** — `/fonts/*.woff2` 요청이 미들웨어에 걸려 `/landing`으로 307 리다이렉트되어 폰트 로드 실패. 기존 매처가 `.svg`만 제외하고 있었음.

```ts
// 수정: 폰트/이미지 확장자 전체를 미들웨어 제외
matcher: ["/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|woff2?|ttf|otf|png|jpg|jpeg|gif|webp|ico)$).*)"],
```

부수 효과: 모든 정적 자산이 요청마다 Supabase 인증 호출(`getUser`)을 타던 낭비 제거.

## 5. 디버깅 기록 (재발 시 참조)

| 증상                      | 원인                                                                                                                 | 판별법                                                                        |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| 수정했는데 옛 폰트가 보임 | 브라우저 디스크 캐시                                                                                                 | `(from disk cache)` 표시 / 시크릿 창 테스트                                   |
| 재시작해도 옛 CSS가 나감  | **`.next` Turbopack 캐시** (2회 재발)                                                                                | curl로 서버 응답 직접 확인 → 디스크 파일과 비교. 해결: `.next` 삭제 후 재시작 |
| 폰트 요청이 안 보임       | DevTools **텍스트 필터에 "font" 입력** — 서브셋 파일명(`PretendardVariable.subset...`)에 "font" 문자열이 없어 걸러짐 | 텍스트 필터 대신 **Font 타입 칩** 사용                                        |
| 폰트만 404                | 미들웨어 매처 (위 §4)                                                                                                | Status 307→404 체인, Initiator 확인                                           |

교훈: **"디스크 파일 ↔ 서버 응답 ↔ 브라우저 표시"는 3개의 다른 층** — 어긋나면 층 사이의 캐시를 의심하고, curl로 브라우저를 배제하면 절반은 좁혀진다.
