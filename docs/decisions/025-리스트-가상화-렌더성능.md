# 리스트 가상화 — 대용량 북마크 렌더 성능

> 북마크가 수백~수천 개일 때 초기 렌더 지연·스크롤 버벅임을 가상 스크롤로 해결한 기록.
> 라이브 데모: `/demo?count=2000` (가상화 ON) vs `/demo?count=2000&virtual=off` (OFF, 비교용)

---

## 문제

북마크 리스트는 반응형 그리드(1/2/3/4열)로 카드를 **전부 DOM에 렌더**한다. 데이터가 적을 땐 문제없지만, 수백~수천 개가 되면:

- **초기 렌더 지연** — 수천 개 카드 + 이미지 DOM을 한 번에 생성
- **스크롤 버벅임** — 스타일/레이아웃 계산 대상이 계속 많음
- **메모리** — DOM 노드가 데이터에 비례해 증가

현재 실사용자 데이터는 적지만, **확장성을 고려한 설계**로 미리 해결한다.

## 가상 스크롤(windowing) 원리

핵심은 **"화면에 보이는 것만 그린다"**.

```
전체 2,000개 (논리적으로 존재)
┌──────────────┐  ← 스크롤 컨테이너
│ [보이는 행들]  │  ← 실제 DOM엔 이 부분 + overscan 몇 개만 존재
└──────────────┘
   ↑ 나머지는 "높이"만 차지 (spacer)해서 스크롤바 위치는 정상
```

- 전체 높이 = `행 수 × 행 높이` 로 계산해 spacer가 차지 → 스크롤바는 실제 크기처럼 동작
- 스크롤 위치(offset)로 **지금 보여야 할 행 index 범위**를 계산 → 그 행만 렌더
- 각 행은 `transform: translateY(offset)` 로 절대 위치에 배치
- 결과: 데이터가 2,000개든 20,000개든 **DOM 노드 수는 일정**

## 구현 결정

**라이브러리: `@tanstack/react-virtual`** (직접 구현 대신)

- 프로덕션급 가상 스크롤은 스크롤 jank·가변 높이·동적 측정 등 엣지가 많아 검증된 라이브러리가 옳은 선택
- 이미 `@tanstack/react-query`를 써서 생태계 일관성 + headless라 스타일 자유

**반응형 그리드는 "행 단위" 가상화**

- 그리드를 그대로 가상화할 순 없어, 뷰포트 너비로 **열 수(1/2/3/4)를 계산**해 카드를 행으로 묶고 **행을 virtualize**
- Tailwind 브레이크포인트(640/1024/1280)와 열 수를 일치시켜 시각적 동일

**window 스크롤 기반(`useWindowVirtualizer`)**

- 기존 페이지가 window 스크롤이라, 내부 스크롤 컨테이너를 만들면 **중첩 스크롤**이 됨 → UX 해침
- window 가상화 + `scrollMargin`(그리드가 페이지 상단에서 떨어진 오프셋)으로 보정

**동적 높이 측정(`measureElement`)**

- 카드 높이가 내용(제목 줄 수·태그)마다 달라, 추정치(첫 페인트)로 그린 뒤 실제 높이를 측정해 보정

**임계값 전환(`VIRTUALIZE_THRESHOLD = 48`)**

- 작은 리스트는 가상화 오버헤드가 오히려 손해 → 48개 초과일 때만 가상화, 그 이하는 일반 그리드

## 대안 비교

| 방식                      | 장점                                                          | 단점                                                 | 선택    |
| ------------------------- | ------------------------------------------------------------- | ---------------------------------------------------- | ------- |
| `@tanstack/react-virtual` | headless·유연, 그리드/가변 높이/동적 측정, react-query 생태계 | 러닝커브                                             | ✅ 채택 |
| `react-window`            | 가볍고 단순                                                   | 가변 높이·그리드 유연성 약함                         |         |
| 페이지네이션/무한스크롤   | 구현 단순                                                     | 한 화면에 다 못 봄, "부드러운 대량 스크롤" 체감 없음 |         |
| 직접 구현                 | 의존성 없음                                                   | 프로덕션급 완성도 리스크 큼(비추)                    |         |

## 측정 (before/after)

> 측정 방법: `/demo?count=2000&virtual=off`(before) vs `/demo?count=2000`(after)를 각각 열고
> Chrome DevTools **Performance**(스크롤 녹화 → dropped frames), **React Profiler**(초기 커밋 시간),
> Elements(DOM 노드 수), Memory로 기록.

| 지표                     | 가상화 OFF (2,000개) | 가상화 ON           | 개선 |
| ------------------------ | -------------------- | ------------------- | ---- |
| 초기 렌더(커밋) 시간     | _측정_               | _측정_              |      |
| DOM 노드 수              | ~2,000 카드          | ~수십 (보이는 행만) |      |
| 스크롤 시 dropped frames | _측정_               | _측정_              |      |
| JS 힙 메모리             | _측정_               | _측정_              |      |

_(수치는 실측 후 채움 — 라이브 데모에서 OFF/ON 토글로 직접 비교 가능)_

## 트레이드오프 / 한계

- **접근성**: 가상 리스트는 보이지 않는 항목이 DOM에 없어 스크린리더 탐색·Ctrl+F가 제한될 수 있음 → 필요 시 `aria-setsize`/`aria-posinset`로 보완(후속 과제)
- **SEO**: 가상화된 목록은 초기 HTML에 전체가 없음 → 이 리스트는 로그인 후 개인 데이터라 SEO 대상 아님(무관)
- **행 높이 편차가 크면** 스크롤 위치 추정 오차 → `measureElement`로 완화
