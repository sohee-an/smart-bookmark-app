# 026 — Vercel 함수 리전 정렬 (iad1 → icn1)

날짜: 2026-07-07

## 문제 발견

Vercel Speed Insights(RUM)에서 데스크톱 P75 기준 이상 수치 감지:

| 지표 | 측정값 (최근 7일, Desktop P75) | 판정              |
| ---- | ------------------------------ | ----------------- |
| RES  | 58점                           | Needs Improvement |
| LCP  | 7.1s                           | Poor              |
| FCP  | 4.43s                          | Poor              |
| TTFB | 1.67s                          | Poor              |
| CLS  | 0                              | Good              |
| INP  | 72ms                           | Good              |

CLS·INP는 정상인데 **로딩 계열 지표(TTFB → FCP → LCP)만 연쇄적으로 나쁜** 패턴 → 렌더링 문제가 아니라 서버 응답 지연이 뿌리라고 가설.

## 진단 과정

표본이 적은 RUM 대신 직접 측정으로 구간별 분해:

1. **정적 페이지** — `/landing` TTFB ~110ms (CDN 캐시 HIT) → 엣지는 정상
2. **미들웨어** — `/` 익명 리다이렉트 ~110–180ms → 미들웨어도 정상
3. **DB 위치** — 한국에서 Supabase까지 13ms → **Supabase는 서울 리전**
4. **함수 위치** — API Route 응답 헤더:

```
x-vercel-id: icn1::iad1::...
              ↑      ↑
      엣지(인천)   함수 실행 리전(미국 동부) ← 원인
```

Vercel 서버리스 함수의 기본 리전이 `iad1`(워싱턴 D.C.)이라, 배포 시 별도 설정이 없으면 모든 SSR/API가 미국에서 실행된다.

## 원인 구조

```
한국 사용자 → 인천 엣지(icn1)
              → 함수: 미국 동부(iad1)          ← 태평양 왕복 ~200ms
                 → 인증 검증: 서울 Supabase     ← 왕복 ~185ms
                 → 데이터 쿼리: 서울 Supabase   ← 왕복 ~185ms × N
              → 응답: 미국 → 한국              ← ~200ms
```

서버 코드와 DB가 지구 반대편에 있어, **DB 왕복이 발생할 때마다 ~185ms가 가산**된다. 로그인 사용자의 SSR 페이지(미들웨어 인증 + 북마크 prefetch)는 왕복 2~3회 → RUM의 TTFB 1.2~1.7s와 정확히 일치.

## 결정

**Vercel Function Region을 `icn1`(서울)로 변경** — Settings → Functions → Function Region.

판단 기준: 함수 리전은 "사용자와 가까운 곳"이 아니라 **"DB와 가까운 곳"**에 둔다. 함수↔DB 왕복은 요청당 N회 발생하지만 사용자↔함수 왕복은 1회뿐이므로, DB 옆에 붙이는 쪽이 항상 이득. 이 서비스는 사용자(한국)와 DB(서울)가 같은 지역이라 둘 다 만족.

- Hobby 플랜은 리전 1개만 선택 가능 → 서울 단일 선택
- 설정 저장만으로는 미적용, **재배포 필요**

## 결과 (실측)

비로그인 API 경로(`/api/semantic-search` 401 응답, 순수 함수 왕복)를 동일 조건에서 각 5~6회 측정:

|      | 변경 전 (iad1)                     | 변경 후 (icn1)     |
| ---- | ---------------------------------- | ------------------ |
| TTFB | 293~332ms                          | **124~142ms** (웜) |
| 하한 | ~300ms (태평양 왕복이 물리적 바닥) | 그 벽 제거         |

로그인 SSR 경로는 DB 왕복당 185ms → ~5ms로 단축되므로 개선 폭이 더 큼 (요청당 400~500ms+ 추정). RUM(Speed Insights) 반영은 트래픽이 쌓이는 수일 후 재확인 예정.

## 교훈

- Vercel 배포 시 함수 리전은 기본값(iad1)을 그대로 두면 안 된다. **DB 리전과 정렬**하는 것이 첫 번째 체크리스트.
- RUM에서 CLS/INP는 좋은데 TTFB/LCP만 나쁘면 프론트 코드가 아니라 인프라 지리를 먼저 의심할 것.
- `x-vercel-id` 헤더(`엣지::함수리전`)로 함수 실행 위치를 바로 확인할 수 있다.
