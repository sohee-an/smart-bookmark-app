# 주니어 → 탄탄한 미드 레벨 로드맵 (2025~2026)

> 경력 2년+, 사수 없음, 올해 안에 미드 수준 도달 목표
> AI 시대 기준으로 작성

---

## 먼저 솔직하게

주니어와 미드의 차이는 **기술 개수**가 아니다.

| 주니어                 | 미드                           |
| ---------------------- | ------------------------------ |
| "이렇게 하면 동작한다" | "왜 이렇게 동작하는지 안다"    |
| 패턴을 따라 쓴다       | 패턴이 왜 생겼는지 안다        |
| 에러를 고친다          | 에러가 왜 났는지 안다          |
| 기능을 구현한다        | 트레이드오프를 설명할 수 있다  |
| AI가 준 코드를 쓴다    | AI가 준 코드의 문제점을 잡는다 |

**사수가 없다는 건 불리하지 않다.**
사수가 있으면 정해진 방향으로 빠르게 가지만, 스스로 고민하는 근육이 덜 생긴다.
사수 없이 2년 이상 버텼다는 건 이미 자기 주도 학습 능력이 있다는 증거다.

지금 필요한 건 **새 기술 추가**가 아니라 **기존 것들의 깊이**다.

## 주니어 → 미드의 핵심 3가지

### 1. "동작한다"에서 "왜 동작하는지"로

지금 당장 이 질문들에 막힘 없이 답할 수 있는가:

**JavaScript / React:**

- `useEffect` 의존성 배열에 함수를 넣으면 왜 무한 루프가 생기는가
- React가 리렌더링을 결정하는 기준은 정확히 무엇인가
- `Promise.all` vs `Promise.allSettled` 차이와 언제 어떤 걸 쓰는가
- 클로저가 뭔지, 이 코드에서 어디서 쓰이는지 설명할 수 있는가

**Next.js:**

- App Router에서 Server Component가 번들에 포함되지 않는 이유
- `fetch` 캐싱이 어떻게 동작하는지, `revalidate`가 정확히 무엇을 하는지
- `use client`를 최대한 아래로 내려야 하는 이유

**네트워크:**

- CORS가 왜 존재하는지, 서버에서 어떻게 처리하는지
- JWT가 왜 안전한지, 왜 안전하지 않을 수 있는지
- HTTP 캐시 헤더(`Cache-Control`, `ETag`)가 어떻게 동작하는지

막히는 게 있으면 그게 지금 파야 할 곳이다.

---

### 2. 트레이드오프를 설명할 수 있는 능력

미드 레벨의 핵심 역량은 **"왜 이 방식을 선택했는가"를 설명하는 것**이다.

예시 — 이 프로젝트의 결정들을 설명할 수 있는가:

```
Q: 왜 localStorage와 Supabase를 Repository 패턴으로 추상화했나?
좋은 답: 저장소가 바뀌어도 비즈니스 로직을 수정하지 않기 위해.
         테스트 시 mock 구현체로 교체가 용이하고,
         React Native 확장 시에도 동일한 인터페이스 재사용 가능.

Q: 왜 임베딩 실패를 Fire-and-forget으로 처리했나?
좋은 답: 임베딩은 부가 기능이고 북마크 저장이 핵심 기능이기 때문.
         임베딩 실패로 저장 자체가 막히면 사용자 경험 훼손.
         트레이드오프: 검색 품질이 일부 저하될 수 있음.
```

이런 설명을 할 수 없으면 코드를 "이해"한 게 아니라 "작성"만 한 것이다.

**연습법**: 내가 짠 코드에서 "왜 이렇게 했지?"라고 스스로 물어보고
3문장으로 설명하는 연습을 매일 한 가지씩 하기.

---

### 3. 다른 사람의 코드를 읽는 능력

주니어는 자기 코드만 쓴다.
미드는 다른 사람의 코드를 읽고, 이슈를 찾고, 개선점을 제안한다.

**연습법:**

- 오픈소스 프로젝트 이슈 탭에서 "good first issue" 찾아서 코드 읽기
- 현재 프로젝트에서 AI가 생성한 코드를 "내가 리뷰어라면?" 관점으로 보기
- PR을 올리기 전에 스스로 리뷰하기: "내가 이 PR을 처음 보는 사람이라면?"

## 월별 집중 영역

### 1~2월: 기반 다지기 — JS/TS 깊이

지금 쓰는 도구의 내부를 이해하는 달.

**JavaScript 깊이:**

- 이벤트 루프, 마이크로태스크 큐 (실제로 그려보기)
- 프로토타입 체인, 클로저, this 바인딩 완전 이해
- 비동기 처리 패턴: async/await 에러 처리의 함정들

**TypeScript 깊이:**

- 제네릭 직접 작성 (단순 `<T>` 넘어서기)
- 조건부 타입 (`T extends U ? X : Y`)
- `infer` 키워드 이해
- 현재 프로젝트 타입들을 더 정확하게 개선해보기

**레퍼런스:**

- You Don't Know JS (무료) — https://github.com/getify/You-Dont-Know-JS
- TypeScript Handbook — https://www.typescriptlang.org/docs/handbook

---

### 3~4월: 시스템 설계 — 아키텍처 사고

"어떻게 짜야 나중에 덜 고생하는가"를 배우는 달.

**배울 것:**

- SOLID 원칙 — 이론이 아니라 현재 코드에서 위반 사례 찾기
- 의존성 주입 (DI) — Repository 패턴이 왜 테스트하기 쉬운지 연결해서 이해
- 이벤트 기반 설계 — 현재 overlay 시스템이 왜 EventEmitter를 쓰는지
- 데이터베이스 설계 — Supabase 스키마에서 인덱스, N+1 문제, RLS 최적화

**실습:**

- 현재 프로젝트에서 결합도가 높은 곳 찾아서 리팩터링
- 변경에 가장 취약한 코드가 어디인지 찾아보기

**레퍼런스:**

- Refactoring (Martin Fowler) — 코드 냄새 찾는 법
- 가상 인터뷰 사이트: https://www.hellointerview.com (시스템 설계)

---

### 5~6월: 테스트 — 자동화된 자신감

테스트를 잘 쓰는 개발자는 리팩터링을 두려워하지 않는다.
그게 생산성의 진짜 차이를 만든다.

**배울 것:**

- 단위 테스트 — 무엇을 테스트해야 하는가 (구현이 아닌 동작)
- 테스트 더블 — mock vs stub vs spy 차이와 각각 언제 쓰는가
- 통합 테스트 — 어디까지 테스트해야 "충분한가"
- TDD 한 사이클 — 테스트 먼저 → 실패 확인 → 구현 → 통과

**현재 프로젝트 적용:**

```
이미 CLAUDE.md에 TDD 프로세스가 있음.
실제로 테스트를 먼저 작성하는 습관 들이기.
BookmarkService 단위 테스트부터 시작.
```

**레퍼런스:**

- Vitest 공식 — https://vitest.dev
- Testing Library — https://testing-library.com
- "테스트하기 어려운 코드"는 설계가 나쁜 코드라는 것 체험하기

---

### 7~8월: 성능 — 측정하고 개선하기

성능 최적화에서 가장 중요한 원칙:
**측정하지 않고 최적화하지 마라.**

**배울 것:**

- Chrome DevTools Performance 탭 읽는 법
- React DevTools Profiler — 어떤 컴포넌트가 얼마나 렌더링되는가
- Lighthouse 점수보다 Core Web Vitals 실제 의미 이해 (LCP, CLS, FID)
- Next.js 번들 분석 (`@next/bundle-analyzer`)
- 데이터베이스 쿼리 성능 — EXPLAIN ANALYZE, 인덱스 동작

**실습:**

- 현재 프로젝트에서 Lighthouse 돌리고 가장 나쁜 지표 하나 개선
- 불필요한 리렌더링 찾아서 메모이제이션 적용 (근데 왜 필요한지 먼저 측정)

---

### 9~10월: AI 엔지니어링 심화

이건 이 시대에 차별점이 된다.

**앞서 분석한 gap 중 현실적으로 구현 가능한 것:**

- RAG 대화형 질의응답 완성 (북마크에 질문하기)
- Function Calling 적용 (AI가 도구 선택)
- LLM Evals 구축 (응답 품질 수치화)
- 스트리밍 응답 적용

이 중 하나를 이 프로젝트에 실제로 붙이기.
"만들어봤다"가 아니라 **"왜 이렇게 설계했는가"를 설명할 수 있는 수준**까지.

---

### 11~12월: 종합 — 설명하는 능력

배운 것을 **글로 쓰거나 말로 설명**하는 달.

설명할 수 없으면 이해한 게 아니다.

**방법:**

- 기술 블로그 3~5개 작성 (구현 일기가 아니라 "왜"를 설명하는 글)
- 포트폴리오 README를 기술 결정 이유 중심으로 재작성
- 이 프로젝트의 아키텍처를 처음 보는 사람에게 10분 안에 설명하는 연습

## AI 시대에 뒤처지지 않는 법

### AI가 대체 못 하는 것에 집중

AI가 잘하는 것:

- 보일러플레이트 생성
- 알려진 패턴 구현
- 코드 설명
- 테스트 케이스 나열

AI가 못 하는 것:

- **"이 상황에서 무엇이 문제인지" 정의하기**
- **트레이드오프 판단 (비즈니스 맥락 포함)**
- **"이 AI 코드가 왜 잘못됐는지" 잡아내기**
- **시스템 전체의 일관성 유지**

지금 이 목록에서 내가 잘 못 하는 것이 배워야 할 것이다.

### AI를 써서 더 빠르게 배우기

```
나쁜 사용: AI가 코드 줌 → 복붙 → 동작 확인 → 끝

좋은 사용: AI가 코드 줌 → "왜 이렇게 했어?" → 설명 들음
           → "다른 방법은?" → 비교 → 트레이드오프 이해
           → 직접 처음부터 작성해봄
```

AI는 속도를 높여주지만, 이해를 건너뛰면 나중에 한계가 온다.

### 매주 해야 하는 것

```
월: 이번 주 이해 안 된 것 하나 골라서 깊게 파기
화~목: 구현 + 의도적으로 "왜"를 메모하며 작업
금: 이번 주 배운 것 3줄로 정리 (블로그나 노션)
```

## 사수 없이 성장하는 현실적인 방법

### 코드 리뷰를 받는 방법

사수가 없어도 리뷰를 받을 수 있다:

1. **오픈소스 PR 올리기** — 아무리 작아도 메인테이너가 리뷰해준다
2. **기술 커뮤니티** — 한국 React 오픈 카카오톡, Discord 서버
3. **AI 코드 리뷰** — "이 코드를 시니어 개발자 관점에서 리뷰해줘" (한계 있지만 유용)
4. **PR을 공개하고 동료에게 요청** — 회사에서 다른 팀원에게도 요청 가능

### 멘토 없이 방향 잡는 법

**좋은 코드를 많이 읽어라:**

- Vercel, tRPC, Radix UI 같은 잘 만들어진 오픈소스 소스 코드
- "이 사람은 왜 이렇게 짰을까" 관점으로 읽기

**기술 리더들 팔로우:**

- Kent C. Dodds (React, 테스트) — https://kentcdodds.com
- Josh Comeau (CSS, Next.js) — https://www.joshwcomeau.com
- Theo (t3.gg, Next.js, 실용주의) — https://t3.gg
- TkDodo (TanStack Query 핵심 기여자) — https://tkdodo.eu/blog

**커뮤니티에서 답변하기:**
가장 빠르게 실력 느는 방법 중 하나는 Stack Overflow나 Discord에서 다른 사람 질문에 답하려고 시도하는 것이다.
답을 모르면 찾으면서 배우고, 맞게 설명할 수 있으면 진짜 이해한 것이다.

## 체크리스트 — 미드 레벨 기준

이것들에 막힘 없이 답할 수 있으면 미드다:

**기술 이해:**

- [ ] 이벤트 루프를 그림으로 설명할 수 있다
- [ ] React 리렌더링 조건을 정확히 말할 수 있다
- [ ] 현재 프로젝트의 인증 흐름을 처음 보는 사람에게 설명할 수 있다
- [ ] Repository 패턴을 쓴 이유를 트레이드오프 포함해서 말할 수 있다

**설계 능력:**

- [ ] 새 기능을 어느 레이어에 넣을지 FSD 기준으로 결정하고 이유를 댈 수 있다
- [ ] "이 코드가 변경에 취약한지" 판단할 수 있다
- [ ] 성능 이슈가 생겼을 때 어디서부터 디버깅할지 안다

**AI 시대 기준:**

- [ ] AI가 생성한 코드에서 문제점을 찾을 수 있다
- [ ] RAG가 뭔지, 이 프로젝트에서 어디까지 구현됐는지 설명할 수 있다
- [ ] 새 AI 도구/기법이 나왔을 때 "언제 쓰는가"를 판단할 수 있다

## 결론

올해 안에 미드에 도달하는 가장 빠른 경로:

```
1. 이미 짠 코드의 "왜"를 설명할 수 있게 되기 (지금 당장)
2. JS/TS 기반 깊이 채우기 (1~2월)
3. 테스트 습관 만들기 (지금 프로젝트에 적용)
4. AI 기법 하나를 처음부터 끝까지 직접 구현 (RAG 완성)
5. 배운 것을 글로 쓰기 (설명 능력 = 이해의 증거)
```

새 기술을 많이 아는 사람보다,
**지금 쓰는 기술을 깊이 이해하고 이유를 설명할 수 있는 사람**이
채용 시장에서 더 강하다.

특히 AI 시대에는, AI가 코드를 생성하는 건 누구나 하니까
"이 코드가 맞는지 판단하는 능력"이 차별점이 된다.
