# 저장과 AI 처리를 분리하면서 배운 것

> 개인 프로젝트에서 500개 북마크를 한 번에 업로드할 때 AI 파이프라인이 폭주하는 문제를 겪었다.
> 원인을 파고들면서 DB 큐의 한계를 직접 이해하고, Redis 기반 큐로 전환한 과정을 정리했다.

---

## 문제의 시작

처음 구조는 단순했다.

```
북마크 저장 요청 → DB 저장 + AI 임베딩 즉시 실행
```

500개를 한 번에 올리면? AI 파이프라인 500개가 동시에 실행된다. API rate limit에 걸리고, 서버는 폭주한다.

직관적인 해결책은 "나눠서 보내기"였지만, 이건 증상 치료에 가깝다. 진짜 문제는 **저장과 AI 처리가 하나의 요청 흐름에 묶여 있다**는 구조였다.

---

## 1차 해결: DB-backed Queue

가장 단순한 해결책으로 DB 테이블을 큐처럼 쓰는 방법을 선택했다.

```sql
CREATE TABLE ai_jobs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  bookmark_id UUID REFERENCES bookmarks(id),
  status TEXT DEFAULT 'pending', -- pending | processing | done | failed
  retry_count INT DEFAULT 0,
  error_message TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

흐름은 이렇다:

```
북마크 저장 → ai_jobs에 INSERT (status: pending)
                      ↓
             Worker가 주기적으로 pending 확인
                      ↓
             pending → processing → done | failed
```

저장과 AI 처리가 분리됐다. 실패해도 `failed`로 기록되니 재처리도 가능해졌다.

### 상태 머신

각 job의 상태 전환은 명확하게 정의했다.

```
pending → processing → done
                    ↘ failed
```

- **pending**: 큐에 등록됨, 미처리
- **processing**: Worker가 현재 처리 중
- **done**: AI 처리 완료
- **failed**: 오류 발생, error_message 기록

---

## DB 큐의 한계를 직접 확인하다

### 한계 1: 폴링으로 인한 DB 부하

Worker가 job을 찾는 방식이 폴링이다.

```
Worker: "pending job 있냐?"  → DB: "없음"
Worker: "pending job 있냐?"  → DB: "없음"
Worker: "pending job 있냐?"  → DB: "있음!" → 처리
```

job이 없어도 계속 쿼리를 날린다. DB는 "없음"을 돌려줄 때도 파싱 → 실행 계획 → 결과 반환 과정을 밟는다. 공짜가 아니다.

Worker 10개가 1초마다 폴링하면 분당 600번 쿼리. 실제 job이 10개뿐이면 590번은 낭비다. 이 낭비가 쌓이면 유저가 북마크를 조회하는 진짜 쿼리가 느려진다.

### 한계 2: Race Condition

Worker를 하나만 쓰면 문제없다. 처리량을 높이려고 여러 개 띄우면 이런 일이 생긴다.

```js
// Worker 5개 동시 실행으로 재현
const results = await Promise.all([
  processNextJob(),
  processNextJob(),
  processNextJob(),
  processNextJob(),
  processNextJob(),
]);
```

pending job이 1개인 상태에서 실행하면:

```
Worker 1: "abc" 발견
Worker 2: "abc" 발견  ← 같은 job
Worker 3: "abc" 발견  ← 같은 job
Worker 4: null
Worker 5: null
```

SELECT와 UPDATE 사이에 gap이 있어서 여러 Worker가 같은 job을 읽어간다. 결과적으로 같은 북마크에 임베딩이 중복 생성된다.

DB 수준에서는 `SELECT FOR UPDATE SKIP LOCKED`로 해결할 수 있다.

```sql
SELECT * FROM ai_jobs
WHERE status = 'pending'
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

한 Worker가 row를 잠그면 다른 Worker는 그 row를 건너뛴다. 동작은 하지만, 이걸 직접 관리해야 한다는 것 자체가 부담이다.

## 2차 해결: Redis 기반 큐로 전환

DB 큐의 두 가지 한계를 구조적으로 해결하기 위해 Redis 기반 큐(Upstash)로 전환했다.

### 폴링 → Push

DB 폴링은 Worker가 계속 물어보는 구조다.

```
Worker → DB → Worker → DB → Worker → DB ...
```

Redis는 반대다. Worker가 구독만 해두면, job이 생기는 순간 Redis가 Worker한테 밀어준다.

```
Worker: "나 구독할게" → 대기
job 등록되는 순간 → Redis가 Worker한테 Push
```

쓸모없는 쿼리가 사라진다.

### Race Condition 구조적 해결

Redis의 `LPUSH` / `BRPOPLPUSH` 명령어는 원자적으로 동작한다. 한 Worker가 job을 가져가는 순간 다른 Worker는 같은 job을 가져갈 수 없다. `SELECT FOR UPDATE` 같은 걸 직접 관리할 필요가 없다.

### 메모리라서 빠른 이유

DB는 데이터를 디스크에 저장한다.

```
DB 읽기: 메모리 확인 → 없으면 디스크 접근 → 느림
Redis 읽기: 메모리에 다 있음 → 바로 반환
```

디스크 접근은 메모리 접근보다 수백~수천 배 느리다. 단, 서버가 꺼지면 데이터가 날아가기 때문에 Redis는 처리하고 사라져도 되는 임시 데이터에 적합하고, 영구 보존이 필요한 건 여전히 DB에 쓴다.

---

## 결과

```
저장 요청 → DB bulk insert → 즉시 응답 반환
                  ↓ (완전히 별개)
            Redis 큐에 job 등록
                  ↓
            Worker가 하나씩 AI 처리
                  ↓
            Supabase Realtime으로 프론트에 상태 전달
```

저장과 AI 처리가 완전히 분리됐다. 500개를 올려도 서버가 폭주하지 않고, AI 처리는 Worker가 순차적으로 소화한다.

---

## 마치며

DB 큐도 충분히 유효한 선택지다. 별도 인프라 없이 빠르게 구현할 수 있고, 트래픽이 크지 않으면 문제없이 동작한다. 다만 Worker를 여러 개 띄우거나 처리량이 늘어나는 상황이 되면 한계가 드러난다.

Kafka 같은 대용량 스트리밍 시스템은 여러 서비스가 같은 이벤트를 각자 독립적으로 구독해야 하는 구조에서 필요해진다. 지금 규모에서는 Redis 큐로 충분하고, 그 판단 자체가 중요하다고 생각한다.
