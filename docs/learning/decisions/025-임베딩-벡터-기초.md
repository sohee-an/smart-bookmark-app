# 임베딩 벡터 기초 정리

## 임베딩 벡터가 뭐야?

텍스트를 **숫자 배열**로 변환한 것.

```
"Next.js 튜토리얼" → [0.023, -0.841, 0.234, 0.109, ...]
                       ← 보통 1536개 숫자 →
```

의미가 비슷한 텍스트는 숫자도 비슷해.  
그래서 "검색어랑 의미가 가까운 북마크"를 찾을 수 있음 → **시맨틱 서치의 핵심**

---

## LLM 직접 검색 vs 임베딩 벡터 검색

|        | LLM 직접            | 임베딩 벡터  |
| ------ | ------------------- | ------------ |
| 정확도 | 더 높음             | 약간 낮음    |
| 속도   | 느림                | 빠름 (0.1초) |
| 비용   | 비쌈 (토큰 많이 씀) | 저렴         |
| 대용량 | 불가능              | 가능         |

### 왜 LLM 직접 검색이 안 되냐?

북마크 1000개 있으면 매 검색마다 1000개를 LLM한테 다 넘겨야 함.  
토큰 비용 폭발 + 느림.

### 실무에서 쓰는 방식 (RAG 패턴)

```
벡터 검색으로 후보 20개 뽑기 (빠름)
→ LLM한테 그 20개만 줘서 최종 랭킹 (정확함)
```

→ smart-bookmark가 이 방향

## 임베딩을 같은 테이블에 넣으면 왜 무거워지냐?

### 크기 비교

```
title: "Next.js 튜토리얼"       → 약 20바이트
url: "https://..."              → 약 50바이트
summary: "Next.js는..."         → 약 200바이트

embedding: [0.023, -0.841, ...] → 1536개 × 4바이트 = 약 6,144바이트
```

**임베딩 하나가 다른 컬럼 전체보다 10배 이상 큼**

### 어디서 문제가 생기냐?

**1. 불필요한 데이터 조회**  
북마크 목록 불러올 때 임베딩은 필요 없는데 같은 테이블이면 매번 딸려옴

**2. 인덱스 크기**  
벡터 인덱스(ivfflat) 자체가 무거워서 테이블 분리하면 관리가 쉬워짐

### 근데 솔직히

포트폴리오 규모에서는 **같은 테이블에 넣어도 당장 문제 없음**  
Supabase 공식 예제도 같은 테이블에 넣음

> "분리하는 이유를 알고 안 한 것" vs "몰라서 안 한 것"  
> 면접에서 완전 다름. 이유를 알고 선택한 거면 OK.
