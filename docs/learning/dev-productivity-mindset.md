# 개발 생산성을 높이는 사고방식

> 작성일: 2026-04-04
> 대상: 개인 프로젝트 + 회사 프로젝트 공통

---

## 핵심 관점

**생산성의 적은 "더 빠르게 타이핑하기"가 아니라 "같은 문제를 두 번 푸는 것"이다.**

두 번 푸는 경우들:

- 어제 짠 코드가 뭔지 내일 다시 파악함
- 같은 버그를 다른 프로젝트에서 또 만남
- 팀원이 내 코드 이해하려고 나한테 질문함
- 배포했는데 뭔가 이상해서 원인 찾는 데 2시간 씀
- 기획이 바뀌었는데 어디 고쳐야 할지 몰라서 전체 뒤짐

**생산성 향상 = 이 반복을 제거하는 것.**

---

## 1. 코드를 짜기 전에 생각하는 방식

### "나중에 읽는 나"를 위해 짜기

코드를 짤 때 독자가 두 명이다.

- 지금의 나 (실행되게 만들기)
- 3개월 후의 나 / 팀원 (이해하게 만들기)

대부분 전자만 생각한다. 후자를 위해 짜면:

- 변수명에 1초 더 쓴다 → 나중에 파악 시간 10분 절약
- 함수를 작게 쪼갠다 → 버그 위치 찾는 시간 절약
- 왜 이렇게 했는지 주석 한 줄 → "이거 왜 이렇게 했지?" 질문 제거

### "변경될 가능성"을 항상 생각하기

코드를 짤 때 이 질문을 습관화:

> **"이 부분이 바뀐다면 어디를 고쳐야 하나? 한 곳인가 여러 곳인가?"**

여러 곳이면 나중에 하나를 빠뜨린다. 그게 버그가 된다.

- API URL이 바뀌면 → 상수 하나만 바꾸면 되는가
- 사용자 권한 로직이 바뀌면 → 한 파일만 보면 되는가
- 에러 메시지 문구가 바뀌면 → 전체 grep 해야 하는가

이 질문을 코드 짜기 전에 하면, 설계가 달라진다.

## 2. 개인 프로젝트에서 생산성 높이는 법

### 결정 피로를 없애기

개인 프로젝트가 느린 가장 큰 이유는 **"이렇게 할까 저렇게 할까"를 매번 새로 고민**하는 것이다.

해결책: 한 번 결정하면 문서화해서 다시 고민 안 하기

```
이 프로젝트의 CLAUDE.md가 바로 그것.
폴더 구조, 네이밍, 레이어 규칙을 한 번 정해두면
매 파일마다 "어디 넣지?" 고민이 없어진다.
```

**개인 프로젝트 생산성 체크리스트:**

- [ ] 폴더 구조 결정 → 한 번만 고민, 문서화
- [ ] 코드 컨벤션 결정 → ESLint/Prettier로 자동화
- [ ] 자주 쓰는 패턴 → 스니펫이나 템플릿으로 만들기
- [ ] 환경 세팅 → 한 번 세팅하면 다음 프로젝트에 복붙

### 완성의 기준을 낮추기

개인 프로젝트가 완성 못 하고 끝나는 이유:
→ "완성"의 기준이 처음부터 너무 높기 때문

**MVP 사고방식:**

```
전체 기능 다 만들기 (X)
→ 핵심 기능 하나가 동작하는 것 (O)
```

이 프로젝트로 보면:

- 북마크 저장 + 조회만 되면 이미 쓸 수 있다
- AI 분석, 시맨틱 서치는 나중에 붙인 것

## 3. 회사 프로젝트에서 생산성 높이는 법

### 맥락 전환 비용을 줄이기

회사에서 생산성이 낮은 진짜 이유는 **집중이 끊기는 것**이다.

- 슬랙 메시지 → 코드 다시 이해 → 10분 낭비
- PR 리뷰 요청 → 내 작업 컨텍스트 잃음
- 미팅 후 → 어디까지 했지?

**실천법:**

- 작업 시작 전 3줄로 오늘 할 것 적기 (Notion, 메모장 어디든)
- 중단됐을 때 "다음에 어디서 시작할지" 한 줄 메모 남기고 자리 뜨기
- 집중 시간 블록 만들기 (오전 2시간은 슬랙 안 봄)

### 팀의 생산성 = 내 코드의 가독성

회사에서는 내 코드가 팀원의 시간에 영향을 준다.
내가 짠 코드를 팀원이 이해하는 데 30분 쓰면, 그게 팀 전체 생산성 손실이다.

**실천법:**

- PR 설명에 "왜 이렇게 했는지" 한 문단 쓰기
- 복잡한 로직에 결정 이유 주석 달기 (`// API가 배열을 반환하지 않을 수 있어서 체크`)
- 리뷰어가 궁금할 것을 PR에서 먼저 답하기

### 반복 작업 목록 만들기

회사에서 3번 이상 같은 걸 하면, 그건 자동화 대상이다.

```
매주 하는 것들:
- 배포 전 체크리스트 수동 확인 → CI/CD 자동화
- 같은 형태의 API 핸들러 계속 복붙 → 코드 제너레이터
- 에러 로그 찾으러 서버 접속 → 모니터링 대시보드
- 스프린트 회고 정리 → 템플릿 + AI 자동 요약
```

## 4. AI를 생산성 도구로 쓰는 구체적인 방법

### 지금 당장 쓸 수 있는 것들

**코드 짜기 전:**

- "이 기능을 구현하는 방법 3가지를 장단점과 함께 알려줘"
- → 직접 고민하는 시간 30분 → AI와 5분으로 단축

**코드 짜는 중:**

- 보일러플레이트는 AI에게 시키고, 핵심 로직에 집중
- 막히면 "이 에러가 왜 나는지" 먼저 AI에게 물어보기

**코드 짠 후:**

- "이 코드에서 빠진 엣지 케이스가 있어?"
- "이 함수 이름이 의도를 잘 표현하는가?"
- → 셀프 코드리뷰를 AI와 함께

**문서화:**

- 복잡한 로직 설명 초안 → AI가 작성 → 내가 검수
- PR 설명 초안 → AI가 diff 보고 작성 → 내가 수정

### AI를 잘 쓰기 위한 습관 하나

**AI에게 질문할 때 배경을 먼저 줘라.**

```
나쁜 질문:
"이 코드 고쳐줘"

좋은 질문:
"Next.js App Router 프로젝트고, FSD 구조를 쓰고 있어.
현재 BookmarkCard 컴포넌트가 클릭 시 상태가 업데이트되는데
리렌더링이 너무 많이 일어나. 원인이 뭔지 봐줘."
```

배경을 줄수록 AI 답변 품질이 올라가고, 반복 질문이 줄어든다.

## 5. 공부 방향

### "속도"보다 "깊이"를 먼저

빠르게 새 기술 여러 개 배우는 것보다,
**하나를 제대로 이해해서 어디에 왜 쓰는지 아는 것**이 생산성에 더 도움이 된다.

예시:

- React를 빠르게 쓸 수 있는 것 (X)
- React가 왜 리렌더링을 일으키는지 아는 것 (O) → 버그 원인 즉시 파악

### 지금 쓰는 기술 스택을 더 깊게 파기

새 프레임워크 배우는 것보다,
지금 쓰는 것에서 "몰랐던 기능"을 발견하는 게 생산성 향상 속도가 빠르다.

이 프로젝트 기준 깊게 팔 것들:
| 기술 | 깊게 파면 얻는 것 |
|------|-----------------|
| Next.js App Router | 캐싱 전략, Streaming, 병렬 라우트 |
| TanStack Query | optimistic update, infinite scroll, cache 무효화 전략 |
| TypeScript | 조건부 타입, infer, 제네릭 활용 |
| Supabase | RLS 고도화, Edge Functions, Realtime 최적화 |

### 읽는 것보다 만드는 것

튜토리얼 10개 읽는 것 = 직접 만든 것 1개

공부 효율 순서:

```
1. 공식 문서 (개념 파악)
2. 작은 실험 코드 (실제로 동작 확인)
3. 현재 프로젝트에 적용
4. 왜 동작하는지 설명 가능한 수준
```

4번에 도달하지 못하면 반쪽 이해다.

## 6. 생산성 향상 실천 체크리스트

### 매일

- [ ] 작업 시작 전 오늘 할 것 3가지 적기
- [ ] 중단될 때 "다음 시작 지점" 메모
- [ ] 막히면 20분 후에 AI 활용 (20분 내 스스로 먼저)

### 매주

- [ ] 이번 주 같은 작업 3번 이상 한 것 없는지 점검
- [ ] 반복됐으면 자동화하거나 템플릿화
- [ ] 팀에서 나한테 같은 질문 반복됐으면 문서화

### 프로젝트 시작할 때

- [ ] 폴더 구조 / 컨벤션 먼저 결정하고 문서화
- [ ] MVP 기능 1가지 정의 (이것만 되면 배포 가능)
- [ ] "변경될 가능성 높은 것" 미리 분리

---

## 결론

생산성은 타이핑 속도나 기술 개수가 아니다.

**같은 문제를 두 번 풀지 않는 시스템을 만드는 것.**

- 고민을 문서화하면 → 다시 고민 안 함
- 패턴을 추상화하면 → 다시 짜지 않음
- 실패를 기록하면 → 다시 실패하지 않음

AI는 이 모든 것을 가속시키는 도구다.
단, AI를 잘 쓰려면 내가 먼저 문제를 명확히 정의할 수 있어야 한다.
문제를 명확히 정의하는 능력 자체가 생산성이다.
