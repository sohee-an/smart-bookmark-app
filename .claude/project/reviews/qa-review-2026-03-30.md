### Critical (3건) — ✅ 전체 처리 완료 (2026-03-30)

**C-1. 초기 로딩 중 empty state 먼저 노출 (CLS)** ✅ 처리완료
`useBookmarks()`가 로딩 중일 때 `data`가 `[]`로 초기화되어, `BookmarkList`가 즉시 "북마크를 추가하세요." 빈 상태를 렌더링합니다. 실제 북마크가 있는 사용자에게도 로딩 시간 동안 빈 상태가 보입니다. `RecentBookmarkSlider`도 `null`을 반환하다가 데이터 도착 후 갑자기 나타나 레이아웃 시프트(CLS)가 발생합니다.

- 파일: `BookmarkList.tsx` L20, `_home-content.tsx` L22, `RecentBookmarkSlider.tsx` L25-27
- 수정: `isLoading` prop 추가 → `BookmarkList` skeleton grid 4개, `RecentBookmarkSlider` skeleton 슬라이더 3개 렌더링. 로딩 확정 후에만 empty state 표시.

**C-2. 재시도 3회 제한이 UI에 전혀 반영되지 않음** ✅ 처리완료
`handleRetry`에서 3회 초과 시 조용히 `return`하지만, `BookmarkCard`의 "다시 시도" 버튼은 이를 전혀 모릅니다. 3회 이후에도 버튼이 활성화되어 있고 클릭해도 아무 일도 일어나지 않습니다 (Nielsen 1번 위반).

- 파일: `_home-content.tsx` L76-86, `BookmarkCard.tsx` L116-146
- 수정: `exhaustedIds` state 추가 → 3회 초과 시 `BookmarkCard`의 "다시 시도" 버튼 대신 "재시도 횟수를 초과했어요." 메시지 표시.

**C-3. `new URL(url).hostname` 런타임 예외** ✅ 처리완료
썸네일 없는 폴백 UI에서 `new URL(url).hostname`을 직접 호출합니다. 잘못된 URL이면 `TypeError`로 카드 전체가 렌더링 불가 상태가 됩니다. `crawl_failed` 카드에서 발생 확률이 가장 높습니다.

- 파일: `BookmarkCard.tsx` L55
- 수정: try/catch로 감싸고 파싱 실패 시 url 원문 fallback 표시.

### High (6건)

**H-1. `BookmarkDetailPanel` ESC 닫기 미구현**
`AddBookmarkOverlay`는 ESC를 처리하는데, 상세 패널은 없습니다. 키보드 사용자가 패널을 닫을 방법이 X 버튼뿐입니다. WCAG 2.1 SC 2.1.1 위반.

**H-2. `BookmarkDetailPanel` 포커스 트랩 미구현**
패널이 열려도 포커스가 이동하지 않고 탭 키로 패널 외부를 계속 탐색할 수 있습니다. 스크린 리더 사용자가 패널 열림 상태를 인식 불가. WCAG 2.1 SC 2.4.3 위반.

**H-3. 저장 실패 시 사용자 피드백 없음**
`handleSave`의 `catch`에서 `console.error`만 실행됩니다. 저장 실패 시 사용자에게 아무 알림이 없습니다. 삭제 실패는 `deleteError` state로 처리하는 것과 불일치.

- 파일: `BookmarkDetailPanel.tsx` L57-68

**H-4. stuck 자동 처리 사용자 인지 불가**
5분 초과 자동 failed 전환이 무음으로 일어납니다. "분석 중"이던 카드가 갑자기 실패 상태로 바뀌는 이유를 사용자가 알 수 없습니다. 추가로 의존성 배열이 `[!!bookmarks.length]`여서 북마크가 이미 있는 상태에서 새 stuck 북마크가 추가돼도 effect가 재실행되지 않습니다.

- 파일: `_home-content.tsx` L30-44

**H-5. 아바타 드롭다운이 `group-hover:block`으로만 열림**
키보드 포커스와 모바일 터치로 접근 불가합니다. 로그아웃의 유일한 진입점인데 키보드/터치 사용자가 사용 불가능합니다. WCAG 2.1 SC 2.1.1 위반.

- 파일: `Header.tsx` L212-234

**H-6. 파이프라인 네트워크 에러 catch가 항상 `failed` 처리**
최상위 `catch`가 크롤 단계 실패도 `aiStatus: "failed"`로 처리합니다. 크롤 실패인데 "AI 요약에 실패했어요"가 표시됩니다. 파이프라인 진행 위치에 따라 `crawl_failed` / `failed`로 분기해야 합니다.

- 파일: `useBookmarkPipeline.ts` L87-91

### 잘된 부분

1. **`BookmarkCard` 상태별 skeleton** — crawling/processing 각 단계에 맞는 `animate-pulse` skeleton을 썸네일/제목/요약 별도 구현. 매우 세밀함.
2. **`crawl_failed` / `failed` 타입 분리** — 에러 메시지까지 구분하여 표시. 사용자에게 정확한 정보 제공.
3. **파이프라인 2단계 낙관적 업데이트** — 크롤 완료 즉시 썸네일/제목 반영, AI 완료 후 summary/tags 반영.
4. **삭제 2단계 확인 플로우** — isDeleting 중 양쪽 버튼 비활성화 + deleteError 인라인 표시까지 완성도 높음.
5. **비회원 5개 제한 UX** — 모달 내 상태 전환으로 로그인 유도 흐름이 자연스러움.
6. **body scroll lock** — 두 오버레이 모두 일관되게 구현되어 있고 cleanup도 올바름.
7. **stuck 자동 처리 아이디어 자체** — 무한대기 버그 근본 해결 접근법이 정확함.
8. **`useBookmarks` 조건부 refetchInterval** — pending 있을 때만 3초 폴링, 완료 시 중단. 효율적.
9. **optimistic update + rollback 패턴** — `useUpdateBookmark`, `useDeleteBookmark` 모두 표준 패턴 정확 구현.
