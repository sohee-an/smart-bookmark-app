
# 모노레포 학습 정리

## 1. 모노레포란?

하나의 레포지토리 안에 여러 패키지(앱, 라이브러리)를 관리하는 방식.

```
my-monorepo/
  apps/
    web/        ← Next.js 앱
    mobile/     ← React Native 앱
  packages/
    ui/         ← 공통 컴포넌트
    utils/      ← 공통 유틸
    types/      ← 공통 타입
```

> 핵심: 빌드/배포는 각 앱이 따로 함. 근데 코드는 공유함.

---

## 2. 모노레포 vs 멀티레포

|           | 모노레포                    | 멀티레포                              |
| --------- | --------------------------- | ------------------------------------- |
| 공유 코드 | 쉬움 (packages/ 한 번 수정) | 번거로움 (publish → 버전업 → install) |
| 권한 관리 | 복잡 (CODEOWNERS로 해결)    | 간단                                  |
| CI 속도   | 빌드 캐싱 없으면 느림       | 각자 독립적                           |
| 타입 공유 | 한 번에 반영                | 버전 맞춰야 함                        |

**모노레포가 맞는 상황**

- 공유 코드가 많을 때
- 앱들이 같은 백엔드, 같은 타입 쓸 때
- 혼자 또는 작은 팀

**멀티레포가 맞는 상황**

- 앱 간 공유 코드가 거의 없을 때
- 팀이 완전히 분리돼서 배포 주기도 다를 때
- 각 앱의 기술 스택이 완전히 다를 때

## 3. pnpm을 쓰는 이유

### 3-1. content-addressable store (디스크 절약)

같은 패키지는 글로벌 store에 딱 한 번만 저장. 나머지는 하드링크로 연결.

```
~/.pnpm-store/react@18/  ← 실제 데이터 1개

프로젝트A/node_modules/react  → 하드링크
프로젝트B/node_modules/react  → 하드링크
프로젝트C/node_modules/react  → 하드링크
```

> 하드링크: 복사가 아니라 같은 데이터를 다른 이름으로 가리키는 것. 디스크 공간 안 늘어남.

### 3-2. 유령 의존성(Phantom Dependency) 차단

npm은 node_modules가 평탄화(hoisting)돼서 내가 설치 안 한 패키지도 import 가능.

```js
// 내 package.json에 B 없음
// 근데 npm에서는 이게 동작해버림
import { something } from "B";
```

이게 문제인 이유: A가 나중에 B 의존성 끊으면 내 코드가 이유도 모르고 터짐.

pnpm은 package.json에 명시 안 한 패키지는 아예 import 못 하게 막음.

## 4. pnpm-workspace.yaml

```yaml
packages:
  - "apps/*"
  - "packages/*"
```

이 설정이 pnpm한테 말하는 것:

> "apps/ 하위 폴더들이랑 packages/ 하위 폴더들을 각각 독립적인 패키지로 인식해라"

이게 없으면 pnpm은 그냥 단일 프로젝트로 봄.
이게 있으면 패키지 간 로컬 의존성 연결 가능.

```json
// apps/web/package.json
{
  "dependencies": {
    "@smart-bookmark/ui": "workspace:*"  ← npm 아니라 로컬에서 찾음
  }
}
```

> `workspace:*` = 로컬 패키지에서 찾겠다는 신호

## 5. .npmrc 설정

```properties
engine-strict=true          ← package.json engines 필드 버전 강제
shamefully-hoist=false      ← 유령 의존성 차단 (pnpm 기본값)
strict-peer-dependencies=false
```

`engine-strict=true` 는 package.json의 engines 필드와 다른 버전이면 설치 자체를 막음.

```json
// package.json
{
  "engines": {
    "node": ">=20.0.0",
    "pnpm": ">=8.0.0"
  }
}
```

## 6. packages/ 구조 (Turborepo 패턴)

```
packages/
  ui/                  ← 공통 React 컴포넌트 (Button, Input 등)
  utils/               ← 플랫폼 무관 순수 로직
  types/               ← 공통 타입 정의
  typescript-config/   ← 공통 tsconfig 베이스
  eslint-config/       ← 공통 ESLint 규칙
```

> 설정 파일도 패키지로 분리하는 이유:
> packages/ui, packages/utils 도 TypeScript, ESLint 씀.
> 각자 설정 따로 만들면 중복. 공통 베이스 패키지 만들고 extends 해서 씀.

## 7. typescript-config 패키지 구조

```
packages/typescript-config/
  package.json
  base.json          ← 모든 패키지 공통 TS 설정
  nextjs.json        ← Next.js 앱 전용 (base extends)
  react-library.json ← React 컴포넌트 패키지 전용 (base extends)
```

```json
// base.json
{
  "compilerOptions": {
    "target": "ES2017",
    "strict": true,
    "esModuleInterop": true,
    "moduleResolution": "bundler"
    // ...
  }
}

// nextjs.json
{
  "extends": "./base.json",
  "compilerOptions": {
    "plugins": [{ "name": "next" }],
    "jsx": "preserve",  // Next.js가 JSX 변환 직접 처리
    "noEmit": true
  }
}

// react-library.json
{
  "extends": "./base.json",
  "compilerOptions": {
    "jsx": "react-jsx"
  }
}
```

각 앱/패키지에서 사용:

```json
// apps/web/tsconfig.json
{
  "extends": "@smart-bookmark/typescript-config/nextjs.json",
  "compilerOptions": {
    "paths": { "@/*": ["./src/*"] }
  }
}

// packages/ui/tsconfig.json
{
  "extends": "@smart-bookmark/typescript-config/react-library.json"
}
```

> `private: true` 를 package.json에 넣는 이유:
> 내부 전용 패키지라 npm에 실수로 배포되면 안 되니까.

## workspace:\*

```json
"@smart-bookmark/eslint-config": "workspace:*",
"@smart-bookmark/typescript-config": "workspace:*"
```

workspace:\* 는 의존성을 npm이 아니라 로컬 모노레포에서 찾겠다는 신호이다.

## 8. Turborepo

**pnpm workspace = 패키지 연결**
**Turborepo = 빌드를 빠르고 똑똑하게**

Turborepo가 하는 일:

1. **의존성 순서 파악** → packages/ui 빌드 끝나야 apps/web 빌드 시작
2. **빌드 캐싱** → 코드 안 바뀌었으면 다시 빌드 안 함
3. **병렬 빌드** → 의존성 없는 패키지들은 동시에 빌드

# 10. 터보 뜯어보기

## 10-1 dependsOn: ["^build"]

````json
{
  "$schema": "https://turborepo.dev/schema.json",
  "ui": "tui",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": ["$TURBO_DEFAULT$", ".env*"],
      "outputs": [".next/**", "!.next/cache/**"]
    },
    "lint": {
      "dependsOn": ["^lint"]
    },
    "check-types": {
      "dependsOn": ["^check-types"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}```

dependsOn: ["^build"] => 이건 ^의존하는 패키지들을 먼저 빌드하라는 뜻이다
예를 들어 apps/web 이 packages/ui 에 의존하면:
^build = packages/ui 빌드 먼저 → 그 다음 apps/web 빌드
^ 없으면 순서 상관없이 병렬로 빌드해요. 근데 packages/ui 빌드 안 된 상태에서 apps/web 빌드하면 에러 난다.


## 10-2 개발에서의 cache: true 의미

dev 에서만 하는 이유는 개발에서는 실시간으로 변경사항을 봐야하니깐 그렇다

## 10-3 persistent: true
dev 서버처럼 계속 실행되는 프로세스라고 Turborepo한테 알려주는 거다.
없으면 Turborepo가 dev 명령어가 끝나길 기다려요. 근데 next dev 는 끝나지 않고 계속 실행된다. persistent: true 없으면 다른 태스크들이 시작을 못 한다.
