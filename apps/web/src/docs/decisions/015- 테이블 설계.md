# smart-bookmark 테이블 설계

## 테이블 구조

```
users
  ↓ 1:N
bookmarks
  ↓ 1:1          ↓ N:M (중간 테이블)
embeddings     bookmark_tags ↔ tags
```

---

## 테이블 상세

### users

| 컬럼       | 타입      | 설명                     |
| ---------- | --------- | ------------------------ |
| id         | uuid      | PK, Supabase Auth와 연동 |
| email      | text      | 이메일                   |
| password   | text      | 비밀번호 (Auth가 관리)   |
| created_at | timestamp | 생성일                   |

> Supabase Auth 쓰면 users 테이블은 Auth가 자동 생성해줌

---

### bookmarks

| 컬럼          | 타입      | 설명                |
| ------------- | --------- | ------------------- |
| id            | uuid      | PK                  |
| user_id       | uuid      | FK → users.id       |
| url           | text      | 북마크 URL          |
| title         | text      | 제목                |
| summary       | text      | AI 요약             |
| status        | text      | unread / read       |
| thumbnail_url | text      | 썸네일              |
| user_memo     | text      | 유저 메모           |
| ai_status     | text      | pending / completed |
| created_at    | timestamp | 생성일              |
| updated_at    | timestamp | 수정일              |

---

### embeddings

| 컬럼        | 타입         | 설명                    |
| ----------- | ------------ | ----------------------- |
| id          | uuid         | PK                      |
| bookmark_id | uuid         | FK → bookmarks.id (1:1) |
| embedding   | vector(1536) | 벡터 데이터             |
| created_at  | timestamp    | 생성일                  |

> pgvector 확장 필요 → Supabase에서 기본 지원

---

### tags

| 컬럼       | 타입      | 설명               |
| ---------- | --------- | ------------------ |
| id         | uuid      | PK                 |
| name       | text      | 태그 이름 (unique) |
| created_at | timestamp | 생성일             |

---

### bookmark_tags (중간 테이블)

| 컬럼        | 타입 | 설명              |
| ----------- | ---- | ----------------- |
| bookmark_id | uuid | FK → bookmarks.id |
| tag_id      | uuid | FK → tags.id      |

> PK = (bookmark_id, tag_id) 복합키

---

## 설계 결정 & 트레이드오프

### tags — 배열 vs 중간 테이블

배열로 저장하면:

```sql
-- 태그별 검색: 풀스캔 발생
SELECT * FROM bookmarks WHERE 'React' = ANY(tags);

-- 통계 쿼리: 모든 배열을 펼쳐야 함
SELECT unnest(tags), COUNT(*) FROM bookmarks GROUP BY unnest(tags);
```

중간 테이블로 저장하면:

```sql
-- 태그별 검색: 인덱스로 바로 접근
SELECT * FROM bookmarks b
JOIN bookmark_tags bt ON b.id = bt.bookmark_id
WHERE bt.tag_id = 3;

-- 통계 쿼리: bookmark_tags만 보면 됨
SELECT tag_id, COUNT(*) FROM bookmark_tags GROUP BY tag_id;
```

**결론:** AI가 자동생성하는 태그 특성상 태그 수가 무한히 늘어날 수 있고, 태그별 필터링과 통계가 필요하므로 중간 테이블 선택

---

### embeddings — 분리 vs 같은 테이블

임베딩 벡터 하나의 크기:

```
1536개 숫자 × 4바이트 = 약 6KB
다른 컬럼 전체 합계 = 약 300바이트
```

분리했을 때 장점:

- 북마크 목록 조회 시 불필요한 벡터 데이터 로드 방지
- 벡터 인덱스(ivfflat) 관리 분리

**결론:** 현재 규모에서는 같은 테이블도 무방하나, 데이터가 늘어날 것을 고려해 분리

---

### 외래키 방향 — 왜 bookmarks가 user_id를 가지나

users에 bookmark_ids 배열로 저장하면:

- 유저 삭제 시 배열도 동시에 업데이트 필요 → 정합성 문제 발생 가능
- 관리 포인트가 두 곳으로 늘어남

**원칙:** 속하는 쪽이 외래키를 가진다 → bookmarks가 user_id 보유

## 주요 개념 정리

### RLS (Row Level Security)

Supabase는 클라이언트에서 DB를 직접 접근할 수 있어서, 설정 안 하면 다른 유저의 데이터도 조회/삭제 가능.
RLS는 DB 레벨에서 "본인 데이터만 접근 가능" 을 강제함.

```sql
-- 현재 로그인한 유저의 북마크만 조회 가능
create policy "bookmarks_select" on bookmarks
for select using (auth.uid() = user_id);
```

### on delete cascade

연결된 테이블의 데이터를 DB가 자동으로 같이 삭제해줌.
없으면 북마크 삭제 시 embeddings, bookmark_tags를 코드에서 직접 삭제해야 하고, 깜빡하면 정합성 문제 발생.

```sql
-- 북마크 삭제 시 연결된 embeddings 자동 삭제
bookmark_id uuid not null references bookmarks(id) on delete cascade
```

### IVFFlat 인덱스 주석 처리 이유

벡터 인덱스는 데이터가 충분히 있어야 의미가 있음.
데이터가 없거나 적을 때 만들면 오히려 검색이 더 느려질 수 있음.
→ 1000건 이상 쌓이면 그때 활성화

```sql
-- 데이터 1000건 이상 쌓이면 주석 해제
-- create index on embeddings using ivfflat (embedding vector_cosine_ops) with (lists = 100);
```
