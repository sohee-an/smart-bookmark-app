# Smart Bookmark — 시니어 개발자 관점 설계 체크리스트

> "URL 넣으면 크롤링해서 AI 써머리 만들어요"는 누구나 말할 수 있다.
> 아래 질문들에 답할 수 있어야 시니어급 설계 능력을 어필할 수 있다.

---

## 1. 신뢰성 (Reliability)

**크롤링은 본질적으로 불안정하다.**

- 사이트마다 User-Agent 차단, robots.txt, bot 감지가 다름
- 같은 URL도 시간대마다 응답이 다를 수 있음 (캐시, A/B 테스트)
- SPA(React, Next.js)로 만든 사이트는 Cheerio로 못 긁음 → JS 실행 필요

```
정적 HTML  → Cheerio OK
SPA        → Cheerio X, Puppeteer 필요
```

**→ 포폴 단계에서 어디까지 커버할지 명확히 선을 그어야 함**

---

## 2. 성능 (Performance)

**크롤링 + AI 분석 파이프라인 예상 소요시간**

```
크롤링    1~3초
AI 분석   1~3초
임베딩    1초
합계      3~7초
```

유저가 URL 넣고 7초 기다리면 이탈함.

**해결 방향**

- 백그라운드 처리 + **낙관적 업데이트** (카드 먼저 보여주고 AI 결과 나중에 채우기)
- 각 카드가 독립적으로 로딩 상태를 가져야 함
- 스켈레톤 UI로 처리 중임을 명확히 표시

---

## 3. 비용 제어 (Cost Control)

**AI 호출이 무제한이면 비용이 폭발할 수 있음**

- 회원은 북마크 저장이 무제한인가?
- 같은 URL 중복 저장 시 AI 재호출하면 낭비
- 크롤링 결과 캐싱 전략 필요

```
같은 URL 재저장 시
  → 캐싱된 크롤링 결과 재사용
  → AI 재호출 없이 임베딩만 복사
```

**면접 예상 질문**

> "사용자가 많아지면 AI API 비용은 어떻게 관리할 건가요?"

---

## 4. 보안 (Security)

**URL 입력은 공격 벡터임**

### SSRF (Server Side Request Forgery)

내부망 URL을 넣으면 내부 서버를 공격할 수 있음

```ts
// 반드시 차단해야 할 IP 대역
// 127.0.0.1, localhost
// 192.168.x.x
// 10.x.x.x
// 169.254.x.x (AWS 메타데이터)
```

### 악성 URL

크롤링 중 악성 스크립트 실행 위험 → Cheerio는 JS 실행 안 하므로 상대적으로 안전하지만 주의 필요

**면접 예상 질문**

> "SSRF 공격은 어떻게 막았나요?"

---

## 5. 데이터 품질 (Data Quality)

**임베딩 품질 = 써머리 품질**

- 써머리가 부정확하면 시맨틱 서치 결과가 엉터리
- AI 모델이 바뀌면 기존 임베딩과 새 임베딩의 벡터 공간이 달라짐

```
모델 변경 시 → 전체 임베딩 재생성 필요 (배치 작업)
```

지금부터 설계에 반영해야 할 것들

- 어떤 임베딩 모델을 쓰는지 DB에 기록
- 모델 버전 컬럼 추가 고려

```ts
{
  embedding,
  embedding_model: 'text-embedding-3-small', // 모델 버전 기록
}
```

**면접 예상 질문**

> "임베딩 모델을 바꾸면 기존 데이터는 어떻게 처리할 건가요?"

---

## 6. 확장성 (Scalability)

**동시에 100명이 URL 넣으면?**

- 크롤링 + AI 호출이 100개 동시에 터짐
- API 한도 초과, 서버 다운 가능성

**해결 방향: 큐(Queue) 시스템**

```
URL 입력
  ↓
큐에 작업 추가
  ↓
워커가 순서대로 처리
  ↓
결과 반환
```

포폴 단계에서 구현 안 해도 되지만
**"확장하려면 BullMQ 같은 큐 시스템이 필요합니다"** 라고 말할 수 있어야 함

---

## 7. 관찰 가능성 (Observability)

**운영하다 보면 어디서 실패하는지 모르면 못 고침**

최소한 이것들은 로깅해야 함

```
크롤링 실패율      → 어떤 사이트에서 자주 실패하는지
AI 호출 성공/실패  → 실패 원인 파악
임베딩 생성 실패   → 시맨틱 서치 안 되는 카드 파악
응답 시간          → 병목 구간 찾기
```

도구 추천

- **Sentry** — 에러 모니터링
- **Supabase 로그** — DB 쿼리 모니터링

---

## 사업적 배포 추가 고려사항

로직 외에 배포 전 반드시 필요한 것들

| 항목                   | 이유                            |
| ---------------------- | ------------------------------- |
| 이용약관               | 없으면 법적으로 문제            |
| 개인정보처리방침       | 유저 데이터 수집 시 필수        |
| 소셜 로그인            | 가입 허들 낮추기 (구글, 카카오) |
| 북마크 export          | 유저 데이터 이동권              |
| 에러 모니터링 (Sentry) | 운영 중 오류 파악               |

## 면접 예상 Q&A 요약

| 질문                          | 핵심 답변                                           |
| ----------------------------- | --------------------------------------------------- |
| SPA 사이트는 어떻게 처리해요? | Cheerio 한계 인지, Puppeteer 폴백 or 범위 제한 명시 |
| SSRF는 막았나요?              | 내부 IP 대역 차단 로직                              |
| AI 비용 관리는?               | 중복 URL 캐싱, 재호출 방지                          |
| 임베딩 모델 바꾸면?           | 배치로 전체 재생성, 모델 버전 컬럼                  |
| 사용자 많아지면?              | 큐 시스템 도입 필요성 인지                          |

배포 전 반드시 (1순위)
이거 없으면 배포 자체가 안 되거나 위험해요
SSRF 차단 → 보안 구멍 뚫린 채로 배포하면 안 됨
중복 URL 캐싱 → 비용 폭발 방지
이용약관/개인정보 → 법적 필수
소셜 로그인 → 가입 허들 낮추기
에러 모니터링 → Sentry, 운영 중 눈 뜨고 있어야 함

배포 후 빠르게 (2순위)
유저 생기기 시작하면 바로 필요해요
크롤링/AI 실패율 로깅 → 어디서 자주 깨지는지 파악
낙관적 업데이트 UX → 느리다는 피드백 나오기 전에
북마크 export → 유저 데이터 이동권
임베딩 모델 버전 기록 → 나중에 모델 바꿀 때 대비

유저 늘어나면 (3순위)
트래픽 생겼을 때 대응
큐 시스템 (BullMQ) → 동시 요청 폭발 대비
SPA 크롤링 (Puppeteer) → 못 긁히는 사이트 민원 나올 때

정리하면
지금 당장 → SSRF + 중복캐싱 + 법적 문서 + Sentry
배포 직후 → 로깅 + UX 개선 + export
나중에 → 큐 + Puppeteer
